## 1 范围与目的

本文档定义车机 App Store 的软件需求，用于指导车机端应用商店、云端分发服务以及运营管理能力的设计、开发、联调与验收。系统目标是在整车安全约束、网络不稳定、车型差异和应用合规要求下，为用户提供可控、可追踪、可升级的应用发现、安装、更新与卸载能力。

### 1.1 目标

- 为车主和乘员提供适配当前车型与账号权益的应用浏览、搜索、安装、更新与卸载能力。
- 为 OEM 运营方提供应用上架、下架、灰度发布、兼容性控制和内容治理能力。
- 在驾驶安全、安装安全、网络异常和存储受限场景下，保证外部可观察行为一致且可恢复。

### 1.2 非目标

- 不描述类设计、线程模型、部署脚本与数据库建表 SQL。
- 不描述应用审核流程中的组织制度和人工审批制度细节。
- 不固化支付流程、订阅计费和第三方开发者门户规则；相关能力在本版本标记为 [待确认]。

## 2 系统背景与边界

### 2.1 系统背景

车机 App Store 是车机生态分发入口，承载车机应用的发现、授权、下载、安装、更新、卸载和运营治理。系统运行于车机主机，依赖车机网络、账号体系、应用安装器、存储管理、驾驶状态信号和云端内容分发平台。

### 2.2 系统职责

- 根据车型、地区、账号与软件版本展示可用应用。
- 管理下载、安装、更新、卸载任务及其用户可见状态。
- 在驾驶中限制高风险操作，并对异常、回滚和下架给出一致反馈。
- 向运营后台回传安装状态、错误事件、应用曝光和更新结果。

### 2.3 系统边界

| 项目 | 本系统责任 | 非本系统责任 |
|------|------------|--------------|
| 应用展示与分发 | 展示应用信息、分发安装包、触发安装/更新/卸载 | 具体业务应用内容实现 |
| 兼容性判断 | 基于车型、区域、OS 版本、屏幕规格做可安装判断 | 业务应用自身功能兼容性修复 |
| 安装执行 | 编排下载、校验、调用系统安装器并展示状态 | 底层包管理器实现 |
| 驾驶安全限制 | 根据驾驶状态限制安装、卸载、复杂交互 | 整车驾驶状态信号产生 |
| 运营治理 | 支持上架、下架、灰度、黑白名单和版本冻结 | 人工审核制度本身 |

## 5 场景摘要

| 场景编号 | 场景名称 | 描述 |
|----------|----------|------|
| UC-01 | 浏览首页 | 用户进入 App Store 首页查看推荐、分类、榜单和活动内容 |
| UC-02 | 搜索应用 | 用户按名称或关键词搜索可安装应用 |
| UC-03 | 查看详情 | 用户查看应用介绍、截图、版本、大小、权限和兼容性说明 |
| UC-04 | 安装应用 | 用户在满足条件时发起应用安装并查看进度 |
| UC-05 | 更新应用 | 用户处理单个应用更新或批量更新 |
| UC-06 | 卸载应用 | 用户从应用详情或已安装列表发起卸载 |
| UC-07 | 网络异常恢复 | 下载过程中网络中断，系统暂停并在网络恢复后继续 |
| UC-08 | 存储不足处理 | 安装前检测到可用空间不足，系统阻止继续并提示处理方式 |
| UC-09 | 驾驶状态限制 | 行驶中用户尝试高风险操作，系统按策略限制并给出明确提示 |
| UC-10 | 并发任务冲突 | 同一应用同时收到安装、更新或下架事件，系统按统一规则处理 |
| UC-11 | 运营下架生效 | 后台下架应用后，车机端停止分发并更新展示状态 |
| UC-12 | 设备重启恢复 | 下载或更新任务在车机重启后恢复或按策略回退 |

| FUN-014 | 系统须支持灰度发布和版本冻结控制 | 不在灰度范围内的设备看不到目标版本；冻结后车机端不得继续推送该版本 | 测试 / 分析 | [工程化补足] |
| FUN-015 | 系统须向用户明确展示应用所需主要权限和敏感能力摘要 | 用户在安装前可查看权限摘要；权限说明与版本信息一致 | 测试 / 检查 | [工程化补足] |
| FUN-016 | 系统须在行驶中限制安装、卸载、登录切换、长文本输入和复杂筛选等高分心操作 | 行驶中触发受限操作时，系统拒绝执行并提示需在安全状态下操作 | 测试 | [工程化补足] |

| FUN-018 | 系统须支持运营后台维护应用元数据、上下架状态、兼容性规则、推荐位和活动配置 | 后台修改生效后，车机端按同步周期或主动刷新机制获取最新内容 | 测试 / 分析 | [工程化补足] |
| FUN-019 | 系统须记录应用曝光、点击、下载、安装、更新、卸载和失败事件，并上传至运营分析系统 | 指定事件可被查询，失败场景带应用 ID、版本和错误码 | 测试 / 检查 | [工程化补足] |
| FUN-020 | 当云端策略、车辆本地策略和任务状态同时作用于同一应用时，系统须按“安全限制优先、下架优先、进行中任务状态可追踪”的规则处理 | 并发场景下最终用户可见状态唯一，不出现同时可安装又已下架的冲突展示 | 测试 / 分析 | [工程化补足] |
| FUN-021 | 系统须支持设备重启后的任务恢复，恢复策略至少覆盖下载中、待安装和更新中的任务 | 重启后任务被恢复、取消或回滚时，系统对用户给出明确状态，不得出现长期未知状态 | 测试 | [工程化补足] |
| FUN-022 | 系统须支持账号登出后的授权刷新和可见内容重算 | 账号变化后，已授权与未授权应用列表及时刷新，已失效授权的安装入口关闭 | 测试 / 分析 | [工程化补足] |

| FUN-024 | 系统须支持付费购买、订阅续费与退款提示 [待确认] | 若启用付费能力，购买入口、订单状态和授权状态一致可追踪 | 测试 / 分析 | [待确认] |

## 7 数据与配置需求

### 7.1 数据需求

| 数据项 | 来源 | 用途 | 生命周期 |
|--------|------|------|----------|
| 应用基础信息 | 运营后台 | 展示名称、图标、简介、分类、开发方等 | 持久化缓存 |
| 应用版本信息 | 发布平台 | 判断可安装版本、更新说明、包大小、校验值 | 持久化缓存 |
| 兼容性规则 | 运营后台 / 策略服务 | 判断当前设备是否可见、可下载、可安装 | 持久化缓存 |
| 车辆画像 | 车机系统 | 车型、区域、OS 版本、屏幕规格、存储信息 | 运行时 |
| 账号与授权信息 | 账号系统 / 权益系统 | 判断登录态与应用可用性 | 运行时 + 短期缓存 |
| 下载与安装任务 | 车机本地任务管理 | 展示进度、恢复状态、失败原因 | 持久化 |
| 埋点与日志事件 | 车机端生成 | 运营分析与故障排查 | 持久化 + 上传 |

## 10 设计约束

| 编号 | 约束 | 原因 |
|------|------|------|
| CON-001 | 车机端安装动作必须复用目标平台现有安装器或包管理能力 | 避免绕过系统安全与权限模型 |

| CON-004 | 云端与端侧之间的应用身份、版本身份和错误码语义必须稳定 | 便于追踪、运营和售后诊断 |
