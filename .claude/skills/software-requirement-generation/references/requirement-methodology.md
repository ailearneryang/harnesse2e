# 软件需求方法论参考

本参考文件沉淀当前目录中的需求编写规则，供需求生成 agent 在写文档时遵循。

## 工作顺序
1. 需求理解与场景识别
2. 功能拆解
3. 验证准则与验证方式编写
4. 数据与配置需求整理
5. 场景、异常与并发冲突补全
6. 可观测性与非功能需求补全
7. 约束、追踪与待确认项整理

## 场景识别
生成需求前必须识别：
- 软件类型：移动端、Web、车机、桌面、嵌入式、小程序等
- 运行环境：OS、硬件、网络、屏幕尺寸
- 目标用户：谁使用，何时使用，在哪用
- 外部依赖：接口、第三方服务、系统能力、设备能力
- 并发来源：多用户、多线程、多任务、多中断、重复点击、重复上报、并行设备访问、异步回调重入

## FR 拆分规则
满足以下任一条件，就必须拆为不同 FR：
- 触发动作不同
- 系统响应不同
- 数据来源不同
- 异常处理不同
- 可以单独开发、联调、测试

一个 FR 中最多描述一个主能力。
若标题中出现“与”“及”“以及”“并支持”等并列能力，应优先检查是否需要拆分。

## 工程化补足判定
### 直接写入正文
- 原始材料已明确表达

### 标记为 [工程化补足]
- 原文未明写，但不补足就无法开发、联调、测试、验收
- 典型包括：超时、重试、空态、异常回退、状态同步、防重复点击、失败后的恢复状态、并发冲突处理规则

### 标记为 [待确认]
- 无法从原始需求和通用场景稳定推断
- 典型包括：精确数值指标、版本下限、容量阈值、并发上限、埋点字段细节

## 并发冲突识别规则
以下任一情况出现时，必须补充并发或竞态相关需求：
- 同一资源可能被两个及以上执行主体同时读写
- 同一动作可能被短时间重复触发
- 同一状态可能被异步回调、定时任务、外部消息同时驱动变更
- 存在前后台并行、主从任务并行、中断与任务并行、云端与端侧并行更新
- 多设备、多会话、多请求可能同时操作同一业务对象

并发需求至少要回答以下问题：
- 冲突对象是什么：状态、资源、记录、配置、命令、文件、会话
- 冲突来源是什么：重复触发、并行写入、重入、乱序返回、延迟回包、抢占
- 系统如何处理：拒绝、排队、覆盖、合并、幂等、重试、回滚、锁定、版本校验
- 用户或调用方可观测到什么结果：提示、错误码、状态保持、最终一致结果
- 验证方式是什么：测试、分析、评审、检查

## 并发冲突编写要求
- 不要求在需求里写锁、信号量、原子变量等实现机制
- 必须写清并发条件下的外部可观察行为
- 如果存在共享状态，必须定义优先级、顺序性或最终生效规则
- 如果存在重复提交，必须定义是否幂等以及重复请求的处理结果
- 如果存在异步回调乱序，必须定义接受旧结果还是丢弃旧结果
- 如果无法判断并发上限或冲突策略，必须标记为 [待确认]

## 验证准则要求
- 每条需求必须包含验证准则和验证方式
- 验证准则必须足够具体，能被测试用例、分析项或评审项直接继承
- 对正常、异常、边界、并发冲突场景，应在需求表或场景摘要中可追踪

## 必备章节
- 范围与目的
- 系统背景与边界
- 参考文档
- 术语与缩略语
- 场景摘要
- 功能需求
- 数据与配置需求
- 非功能需求
- 设计约束
- 验收与追踪
- 待确认问题

## 必查 10 个工程维度
- 并发与竞态控制
- 容错与冗余设计
- 前后台解耦
- 性能指标
- 系统服务集成规范
- 外部接口调用规范
- 资源占用控制
- 埋点与可观测性
- 兼容性要求
- 安全性要求
