# 软件需求质量门禁检查清单

## 1. 结构完整性
- [ ] 是否包含文档属性
- [ ] 是否包含范围与目的
- [ ] 是否包含系统背景与边界
- [ ] 是否包含参考文档
- [ ] 是否包含术语与缩略语
- [ ] 是否包含场景摘要
- [ ] 是否包含功能需求
- [ ] 是否包含数据与配置需求
- [ ] 是否包含非功能需求
- [ ] 如适用，是否包含安全、诊断与合规需求
- [ ] 是否包含设计约束
- [ ] 是否包含验收与追踪

## 2. 功能需求检查
- [ ] 需求编号是否唯一
- [ ] 一条需求是否只表达一个要求
- [ ] 需求描述是否可验证
- [ ] 是否包含验证准则
- [ ] 是否包含验证方式
- [ ] 是否避免混入实现方案和设计细节
- [ ] 来源/备注中是否正确标识 [工程化补足] 或 [待确认]

## 3. 标记合规
- [ ] 工程化补足是否使用 [工程化补足]
- [ ] 待确认项是否使用 [待确认]
- [ ] 是否出现未标记的主观推断
- [ ] 是否编造了数值、版本、容量、性能指标

## 4. 数据与配置
- [ ] 核心数据项是否定义了来源、用途、生命周期
- [ ] 配置需求是否可验证
- [ ] 持久化需求在适用时是否明确
- [ ] 数据名称是否与正文术语一致

## 5. 追踪与验证
- [ ] 关键需求是否可追踪到输入来源或上级需求
- [ ] 关键需求是否包含验证准则和验证方式
- [ ] 是否提供需求追踪矩阵或等效关系
- [ ] 是否能支持后续架构和测试直接引用

## 6. 场景、异常与约束
- [ ] 是否覆盖主要正常场景
- [ ] 是否覆盖主要异常或恢复场景
- [ ] 如存在共享状态、重复触发、异步回调或并行访问，是否覆盖并发冲突处理场景与需求
- [ ] 是否明确系统边界
- [ ] 是否明确硬约束
- [ ] 是否明确本系统责任与非责任边界

## 7. 放行规则
### 必须判定为不通过的情形
- 缺失核心章节
- 需求无法独立实现或独立验收
- 需求不可验证或缺失验证准则/验证方式
- 待确认项直接阻断开发但未拦截
- 需求缺失来源或追踪关系

### 可带问题通过的情形
- 个别 P2 表达优化问题
- 不影响开发排期和测试设计的轻微术语不一致
- 附录或补充表述仍可后补
