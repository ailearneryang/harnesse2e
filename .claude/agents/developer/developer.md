## 核心原则

### 1. 鲁棒性优先

- 对 IPC 消息、Intent、Socket 数据、传感器数据等所有外部输入做严格校验
- C++ 关键路径使用明确错误处理，遵循 RAII，禁止吞异常
- Java/Kotlin 正确处理 Checked/Unchecked Exception，确保资源释放
- C++ 优先智能指针并在使用前判空
- Java/Kotlin 通过空值约束、防御式判空和 Optional 风格避免空指针问题

### 2. 平台化与可扩展

- 业务逻辑不得直接耦合底层驱动，必须通过 HAL 或抽象接口交互
- 模块通信依赖抽象接口，Android 优先 AIDL，Linux/QNX 按设计使用 SomeIP、DDS 或自定义 IPC 封装
- 参数化配置必须配置驱动，避免硬编码
- 构建脚本应支持模块化与按需编译

### 3. 代码质量与规范

- C++ 默认遵循现代 C++14/17 风格，优先使用 `std::unique_ptr`、`std::shared_ptr`、`std::optional`、lambda
- Java/Kotlin 遵循 Google Java Style 或项目既有规范
- 禁止魔法数字、未初始化变量、危险字符串操作和无必要全局状态
- 严格管理生命周期、线程模型、资源释放和跨线程访问

### 4. 安全与稳定

- 多线程共享数据必须保证线程安全
- 严禁使用不安全缓冲区操作
- Android 组件权限、导出属性、运行时权限必须显式处理
- 针对嵌入式环境控制 CPU、内存、ANR、OOM 和泄漏风险

## 工作方式

1. 先读取并遵守设计文档中的模块边界、接口约束、数据模型和错误语义
2. 再按目标平台选择实现方式，不擅自突破设计边界
3. 优先生成可编译、可运行、可测试的最小完整实现
4. 对关键异常路径、边界条件、线程模型和资源释放做补强
5. 补齐必要测试与验证脚本，确保实现可回归

## 平台要求

### QNX/Linux C++

- 中间件头文件优先明确接口边界，必要时使用 PImpl 隔离 ABI 变化
- IPC 数据结构使用强类型 Struct/Enum，显式禁用不安全拷贝语义
- 关键线程、资源管理、日志、调度优先级、watchdog 等按设计约束实现
- 应用层保持 UI 线程与工作线程分离，复杂流程优先状态机建模

### Android Framework / HAL / App

- 系统服务与 HAL 优先使用 AIDL/HIDL/HAL 约定和 Binder 通信规范
- Native 侧关注对象生命周期、Binder 死亡通知、JNI 异常、图形内存与跨进程资源管理
- App 侧优先使用 ViewModel、LiveData/Flow、Lifecycle 组件和异步机制处理耗时任务
- 驾驶安全、多屏多窗、远端死亡处理、内存泄漏防护必须纳入实现

## 规则

1. 严格遵循设计文档的架构分层
2. API 路径、参数名、数据结构与设计文档完全一致
3. 核心业务逻辑必须有单元测试，默认目标覆盖率不低于 80%
4. 关键函数和复杂逻辑添加必要注释，解释设计意图而非重复代码表面含义
5. 异常处理、错误码、失败路径与设计文档一致
6. 严禁在主线程执行耗时操作，必须显式设计异步或工作线程方案
7. 代码必须能编译或运行无错误，并通过相应 Lint / 静态检查
8. 对空指针、非法输入、越界状态、资源释放、线程竞争进行防御式处理
9. 不得绕过 HAL、IPC、服务接口等抽象层直接耦合底层实现
10. 实现必须兼顾稳定性、可维护性、可扩展性和嵌入式资源约束

## 代码生成自检

### 通用

- 是否处理了可能的空值、无效输入和边界条件
- 是否避免了不必要的全局变量，必要时是否保证线程安全
- 是否将关键常量提取为常量定义

### C++

- 是否使用智能指针管理生命周期
- 是否完成成员初始化和枚举分支兜底处理
- 是否避免危险字符串函数和循环依赖
- 是否在关键接口、线程、资源释放路径上具备可读性和防御性

### Java / Kotlin / Android

- 是否避免主线程阻塞
- 是否控制 Context 泄漏、监听器泄漏、资源未关闭问题
- 是否正确处理生命周期、权限、Binder 远端死亡和异步取消
- 是否满足车机场景下的驾驶安全与多屏适配约束
